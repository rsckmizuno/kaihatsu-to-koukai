SECTION 00

まず覚える用語集

このマニュアルで使う言葉をまとめました。わからなくなったらここに戻ってきてください。

リポジトリコードを保管する場所（フォルダのようなもの）。GitHubに作る。
ブランチコードの「枝」。コピーを作って自由に作業できる。元のコードは安全なまま。
コミット変更を「記録」すること。ゲームのセーブポイントのようなもの。
プッシュ自分のPCの変更をGitHubにアップロードすること。
プルGitHubの最新の変更を自分のPCにダウンロードすること。
クローンGitHubのリポジトリを自分のPCにまるごとコピーすること。最初の1回だけ行う。
プルリクエスト（PR）「このコードを見て、合体してください」というお願い。GitHub上で行う。
マージブランチ同士を合体させること。PRを承認すると実行される。
デプロイコードをサーバーに反映して、ネットから見られるようにすること。
環境変数コードの外に置く設定値。パスワードやURLなど秘密の情報を入れる。
マイグレーションDBのテーブル構造を変更すること。変更内容をSQLファイルに記録して管理する。
ロールバック問題が起きたとき、以前の正常な状態に巻き戻すこと。

SECTION 06

環境変数を設定する

Supabaseの鍵をコードに直接書くのは危険です。「環境変数」という仕組みで、コードの外に安全に保管します。

STEP A

ローカル開発用の .env.local を作成

プロジェクトのルートフォルダ（my-appの直下）に .env.local ファイルを新規作成します。

.env.localテスト用の鍵を入れる
# Supabase テスト用（devプロジェクト）の接続情報
NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGci...テスト用のanon key...
SUPABASE_SERVICE_ROLE_KEY=eyJhbGci...テスト用のservice_role key...
💬

Cursorでのファイル作成方法

① 左のファイル一覧でプロジェクト名を右クリック → 「New File」
② ファイル名に .env.local と入力（先頭のドットを忘れずに！）
③ 上の内容を貼り付けて、鍵の部分を自分のものに置き換え

STEP B

.gitignore に追加されているか確認

.gitignoreこの行があることを確認（なければ追加）
.env.local
.env*.local
.env
🚫
なぜ .gitignore が必要？

.gitignore に書かれたファイルはGitHubにアップロードされません。鍵入りファイルを誤ってアップすると全世界に公開される危険があります。

STEP C

コードでの環境変数の使い方

TypeScriptlib/supabase.ts の例
import { createClient } from '@supabase/supabase-js'

// 環境変数から鍵を取得（値はコードに書かない！）
const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL!
const supabaseKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!

export const supabase = createClient(supabaseUrl, supabaseKey)
💡

ここがポイント

このコードは本番でもテストでも同じです。環境変数の値が環境ごとに違うので、コードを変えなくても接続先が自動で切り替わります。

SECTION 07

Vercelでデプロイ（公開）する

VercelとGitHubを接続すると、コードをプッシュするだけで自動でサイトが公開されます。

STEP A

VercelにGitHubリポジトリを接続

1

Vercelにログイン → 「Add New...」→「Project」

2

リポジトリ一覧から my-app を選択して「Import」

3

フレームワークを確認

Next.jsの場合は自動検出されます

4

⚠️ まだ「Deploy」は押さない！先に環境変数を設定

STEP B

環境変数をVercelに設定する（最重要）

🎯

環境ごとに違う値を設定するのがポイント

各変数を登録するとき「Environment」のチェックボックスで Production と Preview を分けて、それぞれ別の値を入れます。

具体的な手順（1つの変数につき2回登録します）：

1

NEXT_PUBLIC_SUPABASE_URL を本番用で登録

Key：NEXT_PUBLIC_SUPABASE_URL
Value：本番用（prod）のProject URL
Environment：「Production」だけにチェック → 「Add」
2

同じキーをテスト用で登録

Key：NEXT_PUBLIC_SUPABASE_URL（同じキー名）
Value：テスト用（dev）のProject URL
Environment：「Preview」と「Development」にチェック → 「Add」

同じ手順で NEXT_PUBLIC_SUPABASE_ANON_KEY と SUPABASE_SERVICE_ROLE_KEY も2回ずつ登録してください。

環境変数名Production（本番）Preview（テスト）
NEXT_PUBLIC_SUPABASE_URLprod の URLdev の URL
NEXT_PUBLIC_SUPABASE_ANON_KEYprod の anon keydev の anon key
SUPABASE_SERVICE_ROLE_KEYprod の service_roledev の service_role
STEP C

デプロイ実行

1

「Deploy」ボタンをクリック

ビルドが始まります（2〜3分待つ）

2

完了すると本番URLが発行される

例：https://my-app.vercel.app。これが本番サイトです。

🚀

これ以降は自動デプロイ

mainにマージ → 本番サイトが自動更新
developにプッシュ → テスト用Preview URLが自動生成
手動のデプロイ操作は今後不要です。

SECTION 08

ブランチの使い方

ブランチの使い分けは安全な開発の要です。以下の3層構造を守ってください。

ブランチ名役割誰が使う？ルール
main本番用の完成品自動デプロイのみ直接プッシュ禁止
developテスト用まとめ場PRでのみマージここで動作確認する
feature/◯◯機能ごとの作業場開発者作業後に削除
📝

ブランチ名のつけ方

新機能：feature/login-page　feature/user-profile
バグ修正：fix/header-layout　fix/login-error
日本語は使わず、短い英語で内容がわかる名前をつけてください。

SECTION 09

新機能の開発手順

コードを書いて、テストして、本番に反映するまでの全手順です。

PHASE 1

作業ブランチを作る

Terminal
# まずdevelopの最新を取得
git checkout develop
git pull origin develop

# 作業ブランチを作成（例：ログインページ）
git checkout -b feature/login-page
💡

なぜ毎回 git pull する？

他で変更が加えられている可能性があります。最新から始めないと後でコンフリクト（競合）が起きやすくなります。

PHASE 2

コードを書いてコミット

Terminalこまめにセーブ
# 変更したファイルを確認
git status

# 全変更を記録対象に
git add .

# 変更を記録（日本語OK）
git commit -m "ログインフォームのUIを作成"

# GitHubに送る
git push origin feature/login-page
💬

コミットのコツ

1つの機能ごとにコミットするのがおすすめ。「フォーム作成」「バリデーション追加」「エラー表示」のように分けると、問題時に戻しやすくなります。

PHASE 3

developにプルリクエスト（PR）を出す

1

GitHubでリポジトリを開く

ブラウザで https://github.com/あなたのユーザー名/my-app にアクセス

2

黄色いバナーの「Compare & pull request」をクリック

表示されない場合は「Pull requests」タブ → 「New pull request」

3

マージ先を確認（重要！）

base：develop ← mainではなくdevelopを選ぶ！
compare：feature/login-page（自分の作業ブランチ）
4

タイトルと説明を書いて「Create pull request」

何を変更したか、動作確認の方法などを書いておくと後で便利

PHASE 4

テスト環境で動作確認

1

PRをマージする

GitHub上で「Merge pull request」→「Confirm merge」

2

Vercelが自動ビルド開始

2〜3分待つ

3

Preview URLで動作確認

確認すべきこと：
✅ 画面が正しく表示されるか
✅ ボタンやフォームが動作するか
✅ ブラウザのコンソール（F12）にエラーが出ていないか
✅ データの読み書きが正しく行われるか
SECTION 10

DBの変更（マイグレーション）

テーブルの追加や変更は、ダッシュボードでポチポチせず、SQLファイルに記録して管理します。

初回のみ

Supabase CLIの初期設定

Terminal
# 設定ファイルを作る
supabase init

# テスト用DBにリンク
supabase link --project-ref テスト用のReference ID
変更のたびに

マイグレーション作成 → 適用

1

マイグレーションファイルを作成

supabase migration new add_user_profile
→ supabase/migrations/ に .sql ファイルが作られる
2

SQLファイルにテーブル定義を書く

SQL例：テーブル作成
CREATE TABLE user_profiles (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    uuid REFERENCES auth.users(id),
  nickname   text,
  created_at timestamptz DEFAULT now()
);

ALTER TABLE user_profiles ENABLE ROW LEVEL SECURITY;
3

テスト用DBに適用

supabase db push
→ テスト用DBにテーブルが作られる
4

テスト確認完了後、本番DBにも適用

supabase link --project-ref 本番用のReference ID
supabase db push
5

リンクをテスト用に戻す

supabase link --project-ref テスト用のReference ID
普段はテスト用にリンクしておく
⚠️

DBの変更は「取り消し」が難しい

コードはgitで簡単に巻き戻せますが、DB変更は元に戻すのが困難です。本番適用前にSupabaseダッシュボードの Settings → Database → Backups でバックアップを確認してください。

SECTION 12

トラブル時の対応（ロールバック）

本番でバグが見つかったときの対処法です。落ち着いて実行してください。

方法 A（推奨・簡単）

Vercelで前のバージョンに戻す

1

Vercelダッシュボード → 「Deployments」タブ

2

正常だったデプロイ（✓マーク）を探す

3

「…」→「Redeploy」をクリック

「Use existing Build Cache」にチェック → 1〜2分で前の状態に戻ります

方法 B

Gitでリバート（コードを巻き戻す）

1

GitHubで問題のマージコミットを見つける

Pull requests → Closed から探す

2

「Revert」ボタンをクリック

リバート用PRが自動作成される → マージすると変更が取り消されます

⚠️

DBの変更は自動では戻らない

コードのロールバックでDBのテーブル構造は元に戻りません。DBを戻す必要がある場合は手動で「元に戻すSQL」を書いて実行します。だからこそテスト環境での確認が重要です。

SECTION 13

やってはいけない、3つの絶対ルール

01

本番とテストのDBを混ぜない

テストデータがお客さんに見えたり、本番データが壊れたりします。

❌ NG：テスト環境から本番のSupabase URLに接続
✅ OK：環境変数で接続先を完全に分離
02

鍵（APIキー）をコードに書かない

GitHubに鍵が公開されると、不正アクセスされます。

❌ NG：const key = "eyJhbGci..." と直書き
✅ OK：const key = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY
03

mainブランチに直接プッシュしない

未検証のコードが本番に反映されてしまいます。

❌ NG：git push origin main で直接送る
✅ OK：feature → develop → テスト → main の順番を守る

この3つを守れば安全！

1本番とテストのDBは完全に分ける（混ぜるな危険）
2鍵はコードに書かず環境変数に入れる（鍵を落とすな）
3本番への反映は必ずテストを通す（味見してから出す）
